# 프로젝트 정의서

문서 작성일: 2026-07-09  
기준 문서: `docs/project-basic-info.md`

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | GTG Solutions & Consult 인터랙션 프로토타입 |
| 패키지명 | `gtg-interaction-prototype` |
| 개발 형태 | 공식 홈페이지 MVP, 프론트엔드 중심 웹앱, 인터랙션 프로토타입 |
| 주요 기술 | Next.js App Router, React, TypeScript, Tailwind CSS, GSAP, Three.js, React Three Fiber, Playwright |
| 현재 문서 상태 | 기획/정의 단계. 일부 콘텐츠와 브랜드 요소는 최종 승인 전 상태 |

### 한 줄 요약

GTG Solutions & Consult의 데이터, 스트리밍, 인프라 자동화, DevOps 품질, 컨설팅 및 기술 지원 역량을 WebGL Hero와 반응형 섹션으로 전달하는 공식 홈페이지 MVP 프로토타입입니다.

### 프로젝트 배경

| 배경 요소 | 설명 |
|---|---|
| 웹사이트 목적 | GTG Solutions & Consult의 주요 서비스 영역을 방문자가 한 페이지에서 빠르게 이해하도록 구성 |
| 레퍼런스 활용 방식 | OVA 레퍼런스의 이미지, 영상, 코드, 문구, 로고, 정확한 배치/타이밍/색상 구성은 복제하지 않고 상호작용 문법만 재해석 |
| 콘텐츠 제약 | 확인되지 않은 GTG 고객사, 성과 수치, 파트너 등급, 인증, 연혁은 임의로 사용하지 않음 |
| 접근성 요구 | WebGL Hero를 사용하더라도 핵심 콘텐츠는 Canvas 밖 semantic HTML로 제공 |
| 검증 요구 | 구현 후 lint, production build, Playwright verification 수행 필요 |

### 프로젝트 목적

- GTG의 핵심 서비스 범위를 공식 홈페이지 첫 화면과 주요 섹션에서 명확하게 전달합니다.
- WebGL 기반 Hero 경험을 제공하되, 모바일, WebGL 미지원, reduced-motion 환경에서도 동일한 핵심 정보를 제공합니다.
- 확인되지 않은 영업/마케팅 주장을 배제하고, 검수 가능한 안전한 콘텐츠 구조를 만듭니다.
- 향후 CMS, 문의 폼, 서비스 상세 페이지, 사례 연구 등으로 확장 가능한 프론트엔드 기반을 마련합니다.

## 2. 해결하려는 문제

### 현재 상황

| 구분 | 현재 상태 |
|---|---|
| 홈페이지 방향 | 공식 홈페이지 MVP 및 인터랙션 프로토타입으로 개발 중 |
| 콘텐츠 상태 | 일부 문구, 브랜드 색상, 로고/파비콘, OG 이미지, 약관 링크 등 최종 승인 필요 |
| 기술 요구 | WebGL Hero, fallback, semantic HTML, reduced-motion, 모바일 대응이 동시에 필요 |
| 검수 요구 | OVA 레퍼런스 복제 방지와 미확인 GTG 사실 사용 금지가 필요 |

### 사용자가 겪는 불편

| 사용자 | 불편 |
|---|---|
| 홈페이지 방문자 | GTG가 제공하는 서비스 범위와 문의 경로를 빠르게 파악하기 어렵다. |
| 잠재 고객 | 데이터 분석, 스트리밍, 자동화, DevOps, 컨설팅 지원 영역이 어떤 관계로 묶이는지 한눈에 이해하기 어렵다. |
| GTG 내부 담당자 | 승인되지 않은 문구나 고객/성과 주장이 섞일 경우 공개 전 검수 부담이 커진다. |
| 개발/QA 담당자 | WebGL, 모바일, fallback, reduced-motion을 일관되게 검증할 기준이 필요하다. |

### 이 문제가 계속될 경우의 리스크

| 리스크 | 영향 |
|---|---|
| 메시지 불명확 | 방문자가 GTG의 핵심 역량을 이해하지 못하고 이탈할 수 있음 |
| 미확인 정보 노출 | 고객사, 인증, 수치 등 검증되지 않은 정보가 공개되어 신뢰 및 법무 리스크 발생 가능 |
| 레퍼런스 유사도 과다 | OVA의 자산, 배치, 타이밍, 색상 구성을 복제한 것으로 보일 수 있음 |
| 접근성 미흡 | WebGL 또는 모션 사용이 어려운 사용자가 핵심 콘텐츠를 이용하지 못할 수 있음 |
| 모바일 경험 저하 | 긴 pinned scroll 또는 3D 효과로 인해 모바일 성능과 가독성이 저하될 수 있음 |

### 이 프로젝트가 해결할 핵심 문제

이 프로젝트는 GTG의 서비스 역량을 “기술적으로 인상적인 Hero 경험”과 “접근 가능한 HTML 콘텐츠”로 동시에 제공하면서, 검증되지 않은 정보와 외부 자산 의존을 배제한 공식 홈페이지 MVP 구조를 만드는 것을 핵심 문제로 정의합니다.

### 정성적 목표

| 목표 | 설명 |

## 7. 성공 기준

### 기능적 성공 기준

| 기준 | 성공 조건 |
|---|---|
| Hero 표시 | 데스크톱에서 WebGL Hero가 빈 화면 없이 렌더링됨 |
| fallback 표시 | WebGL 미지원 또는 강제 fallback 조건에서 HTML/CSS Hero가 정상 표시됨 |
| 모바일 표시 | 모바일에서 핵심 Hero 콘텐츠와 CTA가 가려지지 않음 |
| reduced-motion | `prefers-reduced-motion` 환경에서 정적 Hero와 콘텐츠가 초기 렌더부터 제공됨 |
| 솔루션 섹션 | 5개 솔루션이 모두 표시되고 CTA 링크가 연결됨 |
| 문의 섹션 | 이메일, 전화, 공식 문의 링크가 확인 가능함 |
| semantic HTML | 핵심 서비스/고객/솔루션 정보가 Canvas 외부에서도 접근 가능함 |

### 사용자 경험 기준

| 기준 | 성공 조건 |
|---|---|
| 첫 화면 이해 | 방문자가 Hero에서 GTG 브랜드와 주요 서비스 방향을 인지할 수 있음 |
| 가독성 | 데스크톱과 모바일에서 텍스트 겹침, 잘림, 수평 overflow가 없음 |
| 탐색성 | 내비게이션과 CTA로 주요 섹션 및 문의 영역 이동 가능 |
| 접근성 | 키보드 탐색, semantic structure, reduced-motion 경로가 기본 수준을 충족 |
| 일관성 | 레퍼런스 복제가 아닌 GTG 고유의 구성과 자산으로 보임 |

### 성능 기준

| 기준 | 성공 조건 |
|---|---|
| WebGL 안정성 | canvas nonblank 검증 및 오류 없는 렌더링 |
| 모바일 성능 | 모바일에서는 WebGL/3D 효과를 줄이고 정적 대체 경험 제공 |
| 이미지 로딩 | 외부 이미지 요청 없이 로컬 자산으로만 렌더링 |
| 브라우저 안정성 | 핵심 경로에서 console error 없음 |
| 상세 성능 수치 | LCP, CLS, INP 등 목표값은 확인 필요 |

### 운영 기준

| 기준 | 성공 조건 |
|---|---|
| 빌드 | production build 통과 |
| 린트 | ESLint 통과 |
| E2E 검증 | Playwright verification 통과 |

| 콘텐츠 승인 | 공개 전 승인되지 않은 문구, 고객 표시, 수치, 인증 정보를 제거 또는 확정 |

## 8. 주요 리스크

| 리스크 유형 | 리스크 | 영향 | 대응 방향 |
|---|---|---|---|
| 기술적 리스크 | WebGL이 특정 브라우저/기기에서 실패할 수 있음 | Hero가 빈 화면으로 보일 수 있음 | WebGL 지원 감지, 강제 fallback, Playwright canvas nonblank 검증 |
| 기술적 리스크 | ScrollTrigger, pinned scroll, 3D 효과가 모바일에서 성능 저하를 만들 수 있음 | 모바일 이탈 및 사용성 저하 | 모바일에서는 pinned scroll과 3D 효과 축소, 정적 Hero 사용 |
| 기술적 리스크 | reduced-motion 사용자가 초기 렌더에서 모션을 경험할 수 있음 | 접근성 기준 미달 | 초기 렌더부터 `prefers-reduced-motion` 반영 |
| 일정 리스크 | 최종 copy, 브랜드 색상, OG 이미지, 약관 링크 승인이 늦어질 수 있음 | 공개 일정 지연 | 미정 항목을 별도 체크리스트로 관리하고 임시 draft 상태 유지 |
| 일정 리스크 | WebGL/Playwright 검증 이슈가 반복될 수 있음 | QA 기간 증가 | Desktop, mobile, fallback, reduced-motion 테스트 범위를 조기 고정 |
| 데이터/보안 리스크 | 확인되지 않은 고객명, 로고, 인증, 성과 수치가 노출될 수 있음 | 신뢰 및 법무 리스크 | 공개 승인 여부를 콘텐츠 데이터에 명시하고 미승인 claim 제거 |
| 데이터/보안 리스크 | 외부 이미지 hotlink가 포함될 수 있음 | 저작권, 개인정보, 안정성 리스크 | 로컬 자산만 사용하고 Playwright에서 외부 이미지 요청 검증 |
| 사용자 도입 리스크 | 시각 효과가 강하지만 서비스 메시지가 약할 수 있음 | 방문자가 핵심 내용을 이해하지 못함 | Hero 외부 semantic HTML, 솔루션 텍스트, 문의 CTA를 명확히 유지 |
| 사용자 도입 리스크 | 내부 담당자가 문서 없이 콘텐츠를 수정할 경우 claim-safe 기준이 흔들릴 수 있음 | 유지보수 품질 저하 | 콘텐츠 승인 기준과 금지 claim 목록을 운영 문서로 유지 |
